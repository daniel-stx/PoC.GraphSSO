# Guest Invitation Notes

## What This PoC Covers

- creating a guest invitation through Microsoft Graph
- optionally creating the guest without sending the invitation email immediately
- triggering a later reinvite flow through Graph

## Important Distinction

- `POST /users` is for creating a tenant-local user
- `POST /invitations` is the correct flow for an external guest user
- the `#EXT#` identity is typically generated by Entra as part of the guest invitation process

## Invite Now

If we want Microsoft Graph to send the invitation email immediately, we can use:

```json
{
  "invitedUserEmailAddress": "btzutnpfyhnvnwmezm@fxavaj.com",
  "inviteRedirectUrl": "https://myapp.contoso.com",
  "sendInvitationMessage": true
}
```

This creates the guest invitation and lets Microsoft send the invitation email right away.

## Create Guest Now, Send Later

If we want to create the guest in Entra now but postpone the invitation email, we can use:

```json
{
  "invitedUserEmailAddress": "btzutnpfyhnvnwmezm@fxavaj.com",
  "inviteRedirectUrl": "https://myapp.contoso.com",
  "sendInvitationMessage": false
}
```

This gives us:

- the guest user/invitation flow created in Entra
- no email sent yet
- an `inviteRedeemUrl` that can be stored and used later

## Reinvite Later Through Graph

If we want our background job to ask Graph to trigger a later reinvitation flow, we can use the reinvite endpoint pattern:

```json
{
  "invitedUserId": "00000000-0000-0000-0000-000000000000",
  "invitedUserEmailAddress": "btzutnpfyhnvnwmezm@fxavaj.com",
  "inviteRedirectUrl": "https://myapp.contoso.com",
  "sendInvitationMessage": true
}
```

This is not a "send the original unsent email" action.
It is better understood as:

- create a new Graph invitation/reinvite flow
- reset redemption for the existing guest user
- let Microsoft send the invitation at that later time

## Reinvite Constraint

When reinviting an existing guest, the request should use:

- the same `invitedUserId`
- the same external email address that was used for the original guest invitation

Example:

```json
{
  "invitedUserId": "44b6d204-2160-4a11-8fdc-04cf06659953",
  "invitedUserEmailAddress": "bkfcrgtnqrzehblbjs@nespf.com",
  "inviteRedirectUrl": "https://myapp.contoso.com",
  "sendInvitationMessage": true
}
```

If the reinvite request uses the existing guest user id together with a different external email address, Microsoft Graph can reject the request.

That means this kind of request is likely invalid:

```json
{
  "invitedUserId": "44b6d204-2160-4a11-8fdc-04cf06659953",
  "invitedUserEmailAddress": "different-user@example.com",
  "inviteRedirectUrl": "https://myapp.contoso.com",
  "sendInvitationMessage": true
}
```

This was confirmed during PoC testing. The failure was caused by using the correct guest id with a different invited email than the original guest record.

## Important Data To Persist

If we want to support delayed reinvitation through Graph, we must remember:

- `invitedUserId`
- `invitedUserEmailAddress`
- `inviteRedirectUrl`

The most important item is `invitedUserId`, because the later reinvite flow depends on the existing guest user record.
The persisted `invitedUserEmailAddress` is also important, because reinvite should normally target the same guest email that was originally invited.

## Recommended App Flow

1. Create guest with `sendInvitationMessage: false`.
2. Store `invitedUserId`, `invitedUserEmailAddress`, and `inviteRedirectUrl`.
3. When the scheduled time arrives, background job calls the reinvite flow.
4. Graph sends the new invitation email.

## UX Caveat

After redemption, Microsoft redirects the guest to `inviteRedirectUrl`.
Even if that eventually leads to login, it is often better to redirect first to a small landing page rather than directly into the strict login flow.

Why:

- the user may need onboarding guidance
- the invitation may have just been redeemed
- the flow feels clearer if the page explains what happens next

## Recommendation For This PoC

- use `sendInvitationMessage: false` to prove guest creation without sending
- persist the returned guest user id
- use the reinvite endpoint later to prove that Graph can be told "now is the time"
